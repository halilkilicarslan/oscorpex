# Quality Gates DB Schema and API Contract Implementation Plan

Task ID: H2
Status: implementation plan only
Input design: `docs/quality-gates-center-technical-design.md`

## 1. Scope

This document translates the H1 Quality Gates Center design into database schema, API contracts, permissions, service boundaries, migration order, seed data, and release decision data requirements.

This is not code implementation. It intentionally does not add migrations, repositories, routes, UI, or tests.

## 2. Storage Conventions

| Convention | Decision |
| --- | --- |
| Database | PostgreSQL. |
| ID type | `TEXT` primary keys generated by application services with UUID-compatible values. |
| JSON data | Use `JSONB` for new Quality Gates tables. Existing legacy Control Plane tables that store JSON as `TEXT` remain unchanged until migrated separately. |
| Timestamps | Use `TIMESTAMPTZ NOT NULL DEFAULT now()` for `created_at` and `updated_at` where mutable lifecycle state exists. |
| Immutable evidence | Decision and evidence rows are append-only. They do not receive business updates after insertion except explicit `superseded_at` when a newer row replaces them. |
| Mutable projections | Projection tables, caches, and UI summaries may be mutable. They are not part of this H2 table list. |
| Tenant isolation | All production decision and evidence records require `tenant_id`. A missing `tenant_id` on production quality decisions is default deny. |
| Audit | Every write endpoint and every decision/evidence insert emits an audit event. |
| Existing approvals | Current `approvals` and `approval_events` tables remain legacy Control Plane tables. H2 introduces canonical `approval_requests` and `approval_decisions` for Quality Gates. A compatibility layer can backfill or project legacy approvals later, but the Quality Gates decision engine reads the canonical H2 tables. |

## 3. DB Entity Design

### 3.1 Shared Column Semantics

| Column | Type | Rule |
| --- | --- | --- |
| `id` | `TEXT` | Primary key. |
| `tenant_id` | `TEXT` | References `tenants(id)` when non-null. Required for production decision/evidence records. |
| `project_id` | `TEXT` | References `projects(id)` when the record is project-scoped. |
| `goal_id` | `TEXT` | References `execution_goals(id)` for goal-scoped records. |
| `release_candidate_id` | `TEXT` | References `release_candidates(id)` for release-scoped records. |
| `policy_version` | `TEXT` | Required for gates, evaluations, release decisions, overrides, and rollback triggers. |
| `correlation_id` | `TEXT` | Required for write operations that affect decisions, approvals, overrides, rollbacks, and artifacts. |
| `metadata` | `JSONB` | Optional structured extension field, defaults to `{}`. |
| `created_at` | `TIMESTAMPTZ` | Insert timestamp. |
| `updated_at` | `TIMESTAMPTZ` | Mutable lifecycle update timestamp. |
| `superseded_at` | `TIMESTAMPTZ` | Set only when an immutable row is replaced by newer evidence for the same scope. |

### 3.2 `quality_gates`

Purpose: policy-defined gate configuration used by goal and release evaluations.

| Column | Type | Nullability | Notes |
| --- | --- | --- | --- |
| `id` | `TEXT` | NOT NULL | Primary key. |
| `tenant_id` | `TEXT` | NULL | Null means global default gate policy. Non-null means tenant-specific override. |
| `project_id` | `TEXT` | NULL | Optional project-specific override. |
| `gate_type` | `TEXT` | NOT NULL | Examples: `typecheck`, `security_scan`, `audit_trail_completeness`. |
| `environment` | `TEXT` | NOT NULL | `production`, `staging`, `development`, or `all`. |
| `required` | `BOOLEAN` | NOT NULL | Default false in DB, seed sets production policy. |
| `blocking` | `BOOLEAN` | NOT NULL | Default false in DB, seed sets production policy. |
| `auto_evaluated` | `BOOLEAN` | NOT NULL | Whether evaluator can compute result without human decision. |
| `human_reviewed` | `BOOLEAN` | NOT NULL | Whether human review may be required. |
| `override_allowed` | `BOOLEAN` | NOT NULL | False for non-overridable gates. |
| `override_roles` | `JSONB` | NOT NULL | Array of role keys, default `[]`. |
| `owner_role` | `TEXT` | NOT NULL | Role accountable for policy ownership. |
| `thresholds` | `JSONB` | NOT NULL | Gate-specific thresholds, default `{}`. |
| `status` | `TEXT` | NOT NULL | `active`, `deprecated`, `retired`. |
| `policy_version` | `TEXT` | NOT NULL | Version that introduced this gate policy. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | Default `now()`. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | Default `now()`. |
| `superseded_at` | `TIMESTAMPTZ` | NULL | Set when a newer active gate policy replaces it. |
| `metadata` | `JSONB` | NOT NULL | Default `{}`. |

Primary key: `id`.

Foreign keys:

- `tenant_id -> tenants(id) ON DELETE CASCADE`
- `project_id -> projects(id) ON DELETE CASCADE`

Indexes:

- `idx_quality_gates_lookup` on `(tenant_id, project_id, environment, gate_type, status)`
- `idx_quality_gates_policy` on `(policy_version, status)`
- `idx_quality_gates_type` on `(gate_type)`

Unique constraints:

- `uq_quality_gates_active_scope` unique on `(COALESCE(tenant_id, ''), COALESCE(project_id, ''), environment, gate_type, policy_version)` for non-retired rows.

Tenant rule:

- `tenant_id NULL` is allowed only for global seed policies.
- Tenant-specific gate policy must set `tenant_id`.
- Production decision evaluation must resolve a gate policy from tenant-specific scope or global default; missing tenant on the evaluated decision remains default deny.

Tenant rule:

## 4. Append-only Model Decision

| Record type | Immutable? | Allowed mutation | Reason |
| --- | --- | --- | --- |
| Gate evaluations | Yes | `superseded_at` only. | Evaluations are evidence used by release decisions. |
| Approval decisions | Yes | `superseded_at` only when parent request is superseded. | Human decisions must remain auditable. |
| Release decisions | Yes | `superseded_at` only. | Release decision history must be reconstructable. |
| Override actions | Mostly append-only | Lifecycle state may move `requested -> approved -> active -> expired/revoked/closed`; reason, scope, actor, expiration cannot be changed. | Override has operational lifecycle but original exception evidence is immutable. |
| Rollback triggers | Mostly append-only | Lifecycle state may move to `resolved` or `false-positive`; original trigger payload cannot be changed. | Operational closure is mutable, original trigger is evidence. |
| Artifact references | Yes | `status`, `verified_at`, `updated_at`, `superseded_at` allowed; `location`, `digest`, `produced_by`, `produced_at` immutable after verification. | Artifact identity and digest must remain stable. |

Decision: decision and evidence records are append-only. Mutable projection tables can be introduced later for dashboard speed, but the decision engine must be able to recompute from immutable records.

## 5. Tenant Scope Rules

| Table | Tenant scoped? | Can be global? | Can `tenant_id` be null? | Null condition |
| --- | --- | --- | --- | --- |
| `quality_gates` | Yes for overrides | Yes | Yes | Global default policy seed or platform-wide gate. |
| `quality_gate_evaluations` | Yes | No for production | Only non-production dry-run | Production missing tenant creates default-deny blocker. |
| `approval_requests` | Yes | Admin dry-run only | Only non-production dry-run | Production approval missing tenant is invalid. |
| `approval_decisions` | Yes | No for production | Only if parent request is global dry-run | Must match parent approval tenant. |
| `review_results` | Yes | No for production | Legacy non-production only | Production review gate ignores null tenant. |
| `release_candidates` | Yes | Non-production dry-run only | No for production | Production create fails when tenant missing. |
| `release_decisions` | Yes | Diagnostic dry-run only | No for approved production decisions | Null tenant decision cannot approve production. |
| `override_actions` | Yes | No for production | No for production | Production override request fails when tenant missing. |
| `rollback_triggers` | Yes | Platform critical signal can be global | Only for platform-wide signal intake | Production candidate rollback trigger must tenant-match. |
| `quality_signals` | Optional | Yes | Yes | Platform-wide provider, incident, cost, or audit signal. |
| `artifact_references` | Yes | Public/dry-run only | No for production required artifacts | Production required artifact without tenant is invalid. |

Hard rule: production quality decision records with missing `tenant_id` are default deny. The decision engine must not return `approved` for a production release candidate when candidate, decision, approval, evaluation, override, rollback, or required artifact tenant scope is missing or mismatched.

## 9. Migration Plan

| Step | Migration | Action | Rollback strategy |
| --- | --- | --- | --- |
| 1 | Base enum/type definitions | Add CHECK-compatible text value documentation or DB enum types for gate, approval, release, override, rollback, artifact, signal states. Prefer text with CHECK constraints for easier forward compatibility. | Drop CHECK constraints or enum types only if no dependent tables were created. |
| 2 | Quality gate tables | Create `quality_gates` and `quality_gate_evaluations`. | Drop evaluation table first, then gate table. No production data exists before seed. |
| 3 | Approval tables | Create `approval_requests` and `approval_decisions`. Keep existing `approvals` untouched. | Drop `approval_decisions`, then `approval_requests`. Existing legacy approvals remain available. |
| 4 | Release candidate/decision tables | Create `release_candidates` and `release_decisions`. | Drop `release_decisions`, then `release_candidates`. |
| 5 | Artifact references | Create `artifact_references`; add FK from `quality_signals.artifact_id` later after both tables exist. | Drop dependent FK first, then table if no production artifacts exist. |
| 6 | Rollback/override tables | Create `override_actions` and `rollback_triggers`. | Drop override and rollback tables after disabling routes; immutable audit events remain. |
| 7 | Indexes | Add lookup, latest, status, idempotency, and GIN indexes. Use `CREATE INDEX IF NOT EXISTS`. | Drop indexes by name. |
| 8 | Seed default gates | Insert production default gate policies for required H1 gate taxonomy. | Mark seeded gates `retired` or delete only before any evaluations reference them. After evaluations exist, supersede with a new policy version instead of delete. |

Migration ordering notes:

- `artifact_references` and `quality_signals` have a circular logical relationship. Create both tables without strict FK from artifacts to signals. Add only `quality_signals.artifact_id -> artifact_references(id)`.
- Use idempotent `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ADD COLUMN IF NOT EXISTS`, and named indexes to match current bootstrap style.
- Do not backfill existing legacy `approvals` into `approval_requests` during the first migration. Backfill is a separate compatibility task after API routes are live.

## 11. Decision Engine Data Requirements

The ReleaseDecisionService needs these queries in order:

1. Load release candidate by ID with tenant, project, goal IDs, target environment, state, policy version.
2. Verify tenant access and reject production candidate with missing tenant.
3. Load required gates from `quality_gates` by tenant/project/environment/policy version with global fallback.
4. Load latest non-superseded gate evaluations from `quality_gate_evaluations` by release candidate and gate type.
5. Load valid approvals from `approval_requests` and `approval_decisions` where state/quorum/expiration/tenant/policy version are valid.
6. Load required artifacts from `artifact_references` by goal IDs and release candidate ID, including required type coverage.
7. Load active overrides from `override_actions` where state is `active`, expiration is in the future, and scope matches failed gate evaluation.
8. Load active rollback triggers from `rollback_triggers` where state is `detected`, `validating`, `rollback-required`, or `rollback-recommended`.
9. Load quality signals from `quality_signals` by release candidate, goals, signal type, status, severity, and expiry.
10. Evaluate audit trail completeness using correlation ID, actor, tenant, policy version, artifact digest, approval refs, and gate refs.
11. Insert immutable row into `release_decisions`.
12. Supersede previous active release decision rows for the candidate.
13. Update `release_candidates.state` projection to `blocked`, `approved`, `rollback-triggered`, or keep current state when read-only dry-run.
14. Write audit event `release.decision_evaluated`.

Required query shapes:

```sql
-- Required gates
SELECT *
FROM quality_gates
WHERE status = 'active'
  AND environment IN ($environment, 'all')
  AND policy_version = $policy_version
  AND (
    tenant_id = $tenant_id
    OR tenant_id IS NULL
  )
ORDER BY tenant_id NULLS LAST, project_id NULLS LAST;

-- Latest gate evaluations
SELECT DISTINCT ON (gate_type) *
FROM quality_gate_evaluations
WHERE release_candidate_id = $release_candidate_id
  AND superseded_at IS NULL
ORDER BY gate_type, created_at DESC;

-- Valid approvals
SELECT ar.*, jsonb_agg(ad.*) AS decisions
FROM approval_requests ar
LEFT JOIN approval_decisions ad ON ad.approval_request_id = ar.id
WHERE ar.release_candidate_id = $release_candidate_id
  AND ar.state IN ('pending', 'in-review', 'approved')
  AND ar.expires_at > now()
  AND ar.superseded_at IS NULL
GROUP BY ar.id;

-- Active overrides
SELECT *
FROM override_actions
WHERE release_candidate_id = $release_candidate_id
  AND state = 'active'
  AND expires_at > now()
  AND superseded_at IS NULL;

-- Active rollback triggers
SELECT *
FROM rollback_triggers
WHERE release_candidate_id = $release_candidate_id
  AND state IN ('detected', 'validating', 'rollback-required', 'rollback-recommended')
  AND superseded_at IS NULL;
```
